### For New Engineer
1. Read this entire document.
2. The product is a gamified airdrop scoring platform. The AI Judge is the core mechanism вЂ” it scores Web3 contributions by quality, not volume.
3. The aesthetic is "Arcade RPG" вЂ” neon-on-dark, CRT effects, skill trees, quest cards, loot mechanics. Not a SaaS dashboard.
4. Auth is Web3-first: wallet connect via Dynamic.xyz в†’ signature в†’ Django JWT. Email is secondary.

### Naming Conventions
- Components: PascalCase (`ScoreCard.tsx`, `QuestCard.tsx`)
- Hooks: camelCase with `use` prefix (`useAiJudge.ts`)
- Constants: SCREAMING_SNAKE for values, camelCase for files
- CSS variables: kebab-case (`--gradient-hero`)
- Rarity tiers: lowercase in code (`common`, `rare`, `legendary`), Title Case in UI

### Key Component: AiJudgeDemo

This is the landing page's conversion centerpiece. It must:
1. Accept tweet URL or raw text input
2. Show 3 pre-loaded demo tweets as selectable pills
3. Call the `/api/judge/` Next.js API route (which proxies to Anthropic)
4. Stream the response and animate score bars filling in real-time
5. Display the ScoreCard with animated counters and farming badge
6. Include post-score CTA to join waitlist

The streaming UX is critical вЂ” users should see scores appear progressively, not wait for a loading spinner.

### Django Apps Detail

**accounts** вЂ” Custom User with `wallet_address` as unique identifier. `email` optional. `dynamic_user_id` for Dynamic.xyz link. Web3 verification endpoint.

**judge** вЂ” The AI Judge service. Contains the scoring prompt, Anthropic API wrapper, and response parser. Scoring is async (Celery task). Results cached in Redis (keyed by content hash). Configurable rubric system (JSON schema stored per campaign).

**contributions** вЂ” Stores discovered contributions from crawlers. Fields: `user`, `platform` (twitter/discord/telegram/github), `content_text`, `content_url`, `discovered_at`, `scored_at`, `scores` (JSON: teaching_value, originality, impact), `farming_flag` (genuine/farming/ambiguous), `total_score`, `xp_awarded`.

**profiles** вЂ” User profile aggregation: total XP, per-branch XP, skill tree state (unlocked nodes as JSON), badge collection, rank.

**quests** вЂ” Campaign model: `project` (FK to white-label tenant), `title`, `description`, `difficulty` (D-S), `reward_pool`, `scoring_rubric` (JSON), `start_date`, `end_date`, `max_participants`, `party_size`.

**leaderboard** вЂ” Materialized rankings updated via Celery beat. Global, per-branch, per-quest. Cached in Redis for fast reads.

## Authentication

### Strategy: Web3-First

**Primary auth:** Dynamic.xyz wallet connect в†’ Django JWT
- User connects wallet via Dynamic.xyz SDK on frontend
- Frontend sends wallet_address to `/api/v1/auth/wallet-verify/`
- Django verifies via Dynamic API (or SIWE signature)
- Returns access token (15 min) + refresh token (7 days, httpOnly cookie)
- User model: `wallet_address` is the unique identifier, `email` is optional

**Waitlist auth:** Supabase (no Django involvement)
- Email + optional wallet address в†’ direct Supabase insert
- No JWT needed for waitlist submission

**Dynamic.xyz Setup (Free Tier):**
- Chains: Avalanche C-Chain, Base, (Solana later)
- Wallets: MetaMask, WalletConnect, Coinbase Wallet
- Environment: sandbox (development), live (production)

## Integrations

### Anthropic Claude API (AI Judge)
- **SDK:** `anthropic` Python package
- **Model:** claude-sonnet-4-20250514 (cost-efficient for scoring volume)
- **Pattern:** Celery task queues contribution в†’ builds scoring prompt with rubric в†’ streams response в†’ parses scores в†’ stores in DB в†’ caches result
- **Rate limiting:** Batch scoring during off-peak. Cache by content hash (identical content = identical score).
- **Cost management:** Short, structured scoring prompt. JSON output mode. Sonnet for volume, Opus for appeals/disputes.

### Twitter API v2 (Crawler)
- **Auth:** OAuth 2.0 user context (user grants access)
- **Endpoints:** User tweets, user mentions, search (for replies/threads)
- **Rate limits:** 300 requests/15 min (user context). Celery beat polls every 15 min per user.
- **Storage:** Raw tweet data в†’ Contribution model with platform='twitter'

### The Graph (Subgraph)
- Index on-chain events: PartyCreated, QuestCompleted, RewardDistributed, BadgeMinted
- Existing TG subgraph as starting point
- Deploy to The Graph hosted service (Avalanche + Base)

### Other Integrations
- **Supabase:** Waitlist only. One project for airdrop.works. Table: `waitlist_entries`.
- **Dynamic.xyz:** Free tier. EVM wallets. Env ID in NEXT_PUBLIC_DYNAMIC_ENVIRONMENT_ID.
- **Stripe:** Post-launch (white-label subscriptions). Checkout + webhooks.
- **Resend:** Waitlist confirmation email. 100/day free tier sufficient for launch.
- **Sentry:** Both frontend (@sentry/nextjs) and backend (sentry-sdk[django]).
- **Analytics:** GA4 (NEXT_PUBLIC_GA_ID) + Vercel Analytics (@vercel/analytics).

## Development Workflow

### Git
- Branch: `feature/[description]`, `fix/[description]`
- Commits: conventional (`feat:`, `fix:`, `chore:`)
- Main = production. Dev = staging.
- Solo: self-merge OK. With engineers: PR required for main.

### Commands
```bash
# Frontend
pnpm dev          # Dev server
pnpm build        # Production build
pnpm lint         # ESLint
pnpm type-check   # TypeScript strict check

# Backend
uv run python manage.py runserver    # Dev server
uv run python manage.py migrate      # Run migrations
uv run python manage.py makemigrations # Generate migrations
uv run celery -A config worker -l info # Celery worker
uv run ruff check .                   # Lint
uv run ruff format .                  # Format
```
